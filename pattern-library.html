01
Confirmation
HIGH-STAKES CONFIRMATION GATE
Chat Voice
Trigger Condition
Any irreversible action above defined financial threshold, affecting an external party, or requiring audit record. Examples: invoice approval, PO creation, payment release, vendor onboarding.
Behavioral Definition
AI must restate all critical parameters before executing action. User must provide deliberate, explicit confirmation — not passive continuation. Action must not proceed on ambiguous input.
Decision Logic
IF action.isIrreversible OR
action.amount > threshold OR
action.affectsExternalParty
→ REQUIRE confirmation gate
→ RESTATE: vendor, amount, PO
→ ACCEPT: explicit confirm only
💬 Chat Implementation
Joule
Approving $84,200 to Acme Corp · PO #4420-B · Payment: Nov 14 ACH cycle. Confirm?
User
Yes, approve it
Joule
✓ Approved. Ref: APR-20241107-0847
🎙 Voice Implementation
Joule
Just to confirm — approving eighty-four thousand two hundred dollars to Acme Corp. Say "confirm" or "cancel."
User
Confirm
Joule
Done. Acme Corp approved. Reference number zero eight four seven.
Failure Modes
F1: Accepting "uh-huh", Enter key, or message continuation as confirmation on financial actions — creates accidental approvals.
F2: Not restating critical parameters — user confirms without knowing exact amount.
F3 (Voice): Accepting "yes" or "sure" — mishearing risk on irreversible actions. Keyword "confirm" required.
Required on all actions?
No — standard views/queries skip this
Voice uses keyword?
Yes — "confirm" only, not "yes"
Parameters restated?
Yes — always, before confirmation ask
Cancellable after confirm?
No — communicate this upfront
02
Disclosure
PRE-EXECUTION FLAG SURFACE
Both Channels
Trigger Condition
System detects policy violation, risk signal, data anomaly, or compliance issue before the user requests execution. Flag must be surfaced before — not during or after — the confirmation gate.
Behavioral Definition
AI interrupts the expected flow with the specific issue and specific numbers. User receives full information before making a decision. AI does not proceed silently past a flag.
Decision Logic
IF flag.detected
→ SURFACE before confirmation
→ STATE: exact issue + numbers
→ OFFER: constrained resolution set
→ NEVER proceed silently
💬 Chat — Flag First
Joule
⚠ Before I continue — this invoice ($231K) exceeds PO #5501 limit by $31,000. Choose how to resolve:
1. Request PO amendment
2. Override with reason
3. Reject invoice

If unsure, I can escalate to your controller.
🎙 Voice — Flag First
Joule
Before I continue, I need to flag an issue. This invoice exceeds the PO limit by thirty-one thousand dollars. You can request an amendment, approve with an override, or reject it. Which do you prefer?
Failure Modes
F1: Showing flag only in post-approval audit log — user cannot course-correct and loses trust in AI.
F2: Vague flag language ("there may be an issue") — forces follow-up questions, wastes time.
F3: Surfacing flag after confirmation ask — user already committed mentally, flag feels like a gotcha.
Flag before or after summary?
Before — always pre-confirmation
Specific numbers required?
Yes — never vague ("some overage")
Can user dismiss flag?
Yes — with explicit override + reason
AI proceeds on silent flag?
Never
03
Disclosure
PROGRESSIVE DISCLOSURE LIST
Chat Voice
Trigger Condition
User requests a list of actionable items (invoices, tickets, tasks, requests). List contains 2 or more items. Applies when user must select and act on items sequentially.
Behavioral Definition
Chat: Show full list with inline metadata sorted by priority. Voice: State count and present highest-priority item only — never read the full list aloud. Offer sequential navigation after each resolution.
Decision Logic
Chat:
→ SHOW all items + metadata
→ SORT by priority/urgency

Voice:
→ STATE count ("You have 3...")
→ PRESENT highest priority only
→ OFFER continuation after each
💬 Chat — Full List
Joule
You have 3 invoices pending:
· Acme Corp — $84,200 (7 days overdue)
· TechParts Ltd — $12,500 (due today)
· Global Supply — $231,000 ⚠ over PO limit

Which would you like to review first?
🎙 Voice — One at a Time
Joule
You have three invoices pending. The highest priority is Acme Corp for eighty-four thousand dollars — seven days overdue. Want me to start there?
User
Yes
Joule (after resolution)
Done. Two invoices remaining — want to continue?
Failure Modes
F1 (Voice): Reading all items aloud — exceeds working memory, user asks "can you repeat that?" or abandons.
F2 (Chat): Not showing metadata inline — user must drill into each item to evaluate priority, wastes navigation.
F3: Alphabetical default sort — urgency and financial risk should determine order, not name.
Voice reads full list?
Never — one item at a time
Sort order?
Priority: overdue → flagged → amount
Max voice options per turn?
3 items maximum
Continue prompt after each?
Always — reduce re-entry friction
04
Latency
LATENCY BRIDGE
Voice Only
Trigger Condition
Any voice interaction where API call, data retrieval, or processing is expected to take more than 1.5 seconds. Silence beyond this threshold causes user anxiety, repetition, or call abandonment.
Behavioral Definition
AI produces a bridging phrase immediately after user utterance — before the data is available. Phrase must set expectation of a brief pause without signaling completion or failure. Never use silence as a response.
Decision Logic
IF expected_latency > 1500ms
→ EMIT bridge phrase immediately
→ PAUSE (data retrieval)
→ CONTINUE with result

Bridge phrases:
"I'm pulling that up..."
"Let me check that..."
"One moment..."
🎙 Voice — Latency Bridge in Context
User
What's the status of invoice 1103?
Joule (immediate — 0ms)
I'm pulling that up...
[ 2.1 second API call ]
Joule (after retrieval)
Invoice 1103 from Global Supply is pending approval. It has a flag — the amount exceeds your PO limit. Want me to walk you through the options?
Failure Modes
F1: Silence >2s — user repeats input, creating double-execution risk on action flows.
F2: False completion signal ("Got it!" then pause) — user thinks the action ran before it did.
F3: Over-explaining the delay ("I'm searching our database systems for your invoice...") — verbose, patronizing, creates longer dead air.
Required on voice?
Yes — any call >1.5s
Required on chat?
No — typing indicator sufficient
Bridge phrase length?
3–5 words max
Vary phrases?
Yes — avoid repetitive filler
05
Error
STRUCTURED OVERRIDE WITH AUDIT TRAIL
Both Channels
Trigger Condition
User requests to proceed despite a system-flagged policy constraint. Override is permitted within role authority but requires justification for compliance record.
Behavioral Definition
AI does not block the override — it structures it. Requires: (1) explicit reason from user, (2) reason logged in audit record, (3) relevant stakeholder notified. User is told each of these will happen before they proceed.
Decision Logic
IF user.requests override
→ REQUEST reason (mandatory)
→ CONFIRM: reason will be logged
→ CONFIRM: controller notified
→ EXECUTE only after explicit confirm
→ NEVER silent override
💬 Chat Override Flow
User
Override — we agreed verbally to the PO increase
Joule
To override, I need a brief reason for the audit trail. What should I log?
User
Verbal amendment agreed with procurement 11/7, formal in process
Joule
Approving $231,000 with override. Reason logged. Controller notified. Confirm?
🎙 Voice Override Flow
User
Override
Joule
I'll need a brief reason for the record. Please say it after the tone. [tone]
User
Verbal PO amendment agreed with procurement November seventh
Joule
Approving with override. Your controller will be notified. Say "confirm" to proceed.
Failure Modes
F1: Executing override without requiring reason — compliance gap, no audit trail, user surprised later.
F2: Notifying controller silently without telling user — erodes trust ("did the AI tell my boss?").
F3: Blocking override entirely — pushes decision to untracked channels (email, phone), harder to audit.
Reason required?
Yes — always mandatory
User told of logging?
Yes — before confirmation
Stakeholder notified?
Yes — auto, immediately
Override blocked by AI?
No — structured, never blocked
06
Escalation
GRACEFUL ESCALATION PATH
Chat Voice
Trigger Condition
Task exceeds AI authority. User expresses uncertainty. 2+ recovery attempts fail. User explicitly requests human. Any compliance decision requiring human judgment.
Behavioral Definition
Chat: Escalation always visible as a persistent option during flagged flows. Voice: Offered proactively if user declines offered options or says "I'm not sure." Never buried in a help menu.
Decision Logic
Chat: SHOW escalation as 4th option
on all flagged flows (always visible)

Voice: OFFER escalation if:
→ user says "not sure" / hesitates
→ user declines all options
→ 2+ failed recoveries
→ authority threshold exceeded
💬 Chat — Always Visible
Joule
This invoice exceeds your PO limit. Choose a resolution:
1. Request PO amendment
2. Override with reason
3. Reject invoice
4. Escalate to controller →
🎙 Voice — Triggered by Hesitation
User
I'm not sure what to do here
Joule
No problem — I can loop in your controller Sarah Chen to review this. Want me to send her a summary and flag it for her attention?
Failure Modes
F1: Escalation only in help/settings menu — user gives up before finding it, task abandoned or moved off-system.
F2 (Voice): Offering escalation on every turn — creates noise, undermines AI confidence signal.
F3: Escalation without context transfer — controller receives no summary, user must re-explain everything.
Chat: always visible?
Yes — on all flagged flows
Voice: proactively offered?
Only on hesitation/failure signals
Context transferred?
Yes — summary auto-generated
User told who receives?
Yes — name the human explicitly
07
Clarification
AMBIGUOUS INPUT RESOLUTION
Both Channels
Trigger Condition
User input matches multiple possible intents or entities, or is underspecified for the required action. AI cannot proceed without disambiguation — wrong guess has cost (wrong invoice approved).
Behavioral Definition
AI presents the top 2–3 candidate interpretations as explicit options, not open questions. Never guesses silently on high-stakes actions. On low-stakes queries, can proceed with best guess + offer correction.
Decision Logic
IF input.matchCount > 1
AND action.isHighStakes
→ PRESENT top 2-3 interpretations
→ ASK user to select

IF action.isLowStakes
→ PROCEED with best match
→ STATE assumption explicitly
→ OFFER correction
💬 Chat — Ambiguous Vendor
User
Approve the Global invoice
Joule
I found two matching invoices:
· Global Supply — $231,000 (flagged)
· Global Logistics — $14,800 (clear)

Which did you mean?
🎙 Voice — Ambiguous Vendor
User
Approve the Global invoice
Joule
I found two — Global Supply for two hundred thirty-one thousand, or Global Logistics for fourteen thousand eight hundred. Which one?
Failure Modes
F1: Silent best-guess on high-stakes action — wrong invoice approved, financial error, compliance issue.
F2: Open question ("Which invoice did you mean?") without surfacing options — forces user to recall exact name or amount.
F3: Asking for clarification on low-stakes queries — unnecessary friction, makes AI feel slow and bureaucratic.
Silent guess on high-stakes?
Never
Max options presented?
3 (chat) / 2 (voice)
Low-stakes: proceed + state?
Yes — with correction offer
Ask before showing options?
No — show options directly

Error
SYSTEM ERROR RECOVERY
Both Channels
Trigger Condition
Backend system unavailable, API timeout, data retrieval failure, or partial data returned. AI cannot complete the requested task due to a technical failure outside the user's control.
Behavioral Definition
AI acknowledges the failure honestly, explains what it tried to do, and gives the user a clear next step. Never pretends the task completed. Never gives a generic "something went wrong" with no path forward.
Decision Logic
IF system.error
→ ACKNOWLEDGE: be specific
→ STATE: what was attempted
→ OFFER: retry / alternate path
→ NEVER: false success signal
→ NEVER: "please try again" only
💬 Chat — API Timeout
Joule
I wasn't able to retrieve the invoice details — SAP S/4HANA isn't responding right now. Your approval was not submitted.

You can try again in a moment, or access the invoice directly in the Fiori launchpad. Want me to try again?
🎙 Voice — API Timeout
Joule
I'm having trouble connecting to the system right now. I haven't submitted anything. You can try again in a moment — want me to retry?
Failure Modes
F1: Generic "something went wrong" with no specificity — user doesn't know if action ran or not, calls support.
F2: False success signal on partial completion — user believes invoice was approved, payment missed.
F3: "Please try again later" with no path — user abandons task entirely, moves to manual process.
Confirm action NOT taken?
Yes — explicitly, always
Name the failing system?
If known — helps user triage
Offer alternate path?
Always — never dead end
Retry without user prompt?
No — ask first, user controls

GOVERNING PRINCIPLES

The rules that determine when and how to apply these patterns — and when not to.

01
Channel-native, not channel-adapted
Voice flows are designed from scratch for voice cognition. Never adapt a chat flow for voice by removing visual elements. The medium changes the design fundamentally — timing, information density, confirmation patterns, and option limits are all different.
02
Failure is a first-class design surface
Every pattern must define its failure modes explicitly. An undesigned error state will produce a worse default than a designed one. "We'll handle that later" is a product decision that ships user-facing failures.
03
AI authority has explicit boundaries
Define what the AI can do, what it flags but allows, and what it cannot do at all — before designing flows. These boundaries are not the AI's job to infer. They belong in the system prompt and the pattern spec.
04
Constrain options, don't open them
Open questions ("What would you like to do?") have lower completion rates than constrained menus. Enterprise users under time pressure need a decision set — not a blank field. When in doubt, provide 3 named options.
05
Prompt layer ≠ UI layer
Prompts control language and decision logic. The design system controls visual indicators (icons, badges, colors). Never let the model generate its own UI elements — they will be inconsistent and break when the system changes.
06
Trust is designed, not assumed
Users trust AI systems that tell them what they're doing, what was logged, who was notified, and what happens next. Transparency about AI actions — especially irreversible ones — is a design requirement, not a nice-to-have.
For Product Teams
HOW TO USE THIS LIBRARY
01
Identify your trigger
Match your use case to a pattern's trigger condition. Multiple patterns may apply to a single flow — stack them deliberately.
02
Check your channel
Patterns tagged "Both" still have channel-specific implementations. Read both columns — do not default to the chat version for voice.
03
Review failure modes first
Before implementing, read the failure modes section. These are the most common places where teams deviate from the pattern and introduce user-facing bugs.
04
Adapt, don't override
Patterns can be adapted for context — but changes to trigger conditions or decision logic must be documented. Silent deviations make the library unreliable for other teams.
