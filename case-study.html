The Task & Design Challenge

Enterprise invoice approval is a high-stakes, multi-step workflow. Errors have real financial consequences. This case study designs the same task across two fundamentally different channels — chat and voice — demonstrating that good conversation design is not channel adaptation but channel-native thinking.

Primary User
AP Manager
Finance org, SAP power user, time-pressured
Task Complexity
High Stakes
Financial action, irreversible approval, audit trail required
AI System
LLM-Powered
Joule agent — not rule-based, handles ambiguity
Design Scope
Full Flow
Intent → confirm → execute → error → escalate

Scenario

Maria is an AP Manager at a manufacturing company. She has 3 invoices pending approval over $50K. She's in back-to-back meetings. Between meetings, she uses the SAP mobile app (chat) to approve from her desk, and Joule via phone (voice) while walking between buildings.

Design Principles Governing This Work

1
Channel-native, not channel-adapted
Voice flows are designed from scratch for voice cognition — not adapted from chat. Different confirmation patterns, different error recovery, different information density.
2
High-stakes = explicit confirmation gates
Financial approvals require confirmation architecture. We define exactly what triggers a gate, what information must be spoken aloud vs. surfaced visually, and when to refuse to proceed.
3
Failure is a first-class design surface
Every flow state has a defined error path. Ambiguity, partial match, system error, and user hesitation are all designed — not left to defaults.
4
Human in the loop is a design decision, not a fallback
We define explicitly when AI should defer to human judgment, surface escalation paths proactively, and avoid false confidence.

Chat-Specific Design Decisions

Information Density
Rich inline cards
Invoice details, PO status, and flags rendered in single message. No follow-up questions needed for standard approvals.
Confirmation Pattern
Summary + explicit yes
Restates all critical data before action. Requires deliberate "yes" — not Enter key or continuation.
Error Recovery
Action menu on flag
When a flag surfaces, provides 4 explicit options rather than leaving user to figure out next step.
Memory Use
Session context
Remembers all 3 invoices for session. Tracks which are done. Offers to continue without re-querying.

Voice Flow — SAP Joule (Phone / IVR)

Voice is linear, time-bound, and screenless. Working memory is limited — users cannot scroll back. Every utterance must be self-contained. Confirmation must be spoken, not scanned. Latency creates anxiety. Design must account for all of this from the first word.

HAPPY PATH — Voice-Native Approval
Maria (voice)
Hey Joule, what invoices do I need to approve?
Joule (voice)
You have three invoices pending. The highest priority is Acme Corp for eighty-four thousand two hundred dollars — it's seven days overdue. Want me to start there?
✦ ONE invoice at a time. Highest priority first. Binary next step — reduces cognitive load vs. chat's list.
Maria
Yes
Joule
Acme Corp, invoice ending in 0847, eighty-four thousand two hundred dollars. Purchase order confirmed. Goods received. No disputes on record. Ready to approve?
✦ Key fields read aloud in deliberate sequence. No visual card — all critical info spoken. Short sentences. "Ready to approve?" closes the loop clearly.
Maria
Approve it
Joule
Just to confirm — approving eighty-four thousand two hundred dollars to Acme Corp. Say "confirm" or "cancel."
✦ CRITICAL DIVERGENCE: Voice uses a mandatory spoken confirmation word — not a yes/no question. "Confirm" vs "cancel" prevents mishearing. Required for irreversible financial actions.
Maria
Confirm
Joule
Done. Acme Corp approved. Reference number zero eight four seven. You have two invoices remaining — want to continue?

Voice-Specific Design Decisions

Information Chunking
One item at a time
Never read all 3 invoices upfront. Present highest priority, resolve, then offer to continue.
Confirmation Word
"Confirm" keyword
Explicit keyword required for financial approvals. Prevents "yes/yeah/sure" ambiguity on critical irreversible actions.
Latency Handling
Filler + bridging
"Pulling that up..." fills acoustic silence. 2s+ pauses without filler cause user anxiety and re-prompting.
Option Limit
Max 3 choices
Voice working memory: 3 items maximum. Chat can show 4 buttons. Voice cannot.

Channel Divergence Analysis

The most important design document in this case study. These are not stylistic preferences — they are cognitive and technical constraints that force different design decisions. Each divergence is deliberate.

Design Dimension 💬 Chat 🎙 Voice
Invoice list presentation All 3 shown upfront as a scannable list with status, amount, and flags inline Highest priority only, stated first. Others offered as continuation after resolution
Confirmation pattern Summary card + "Yes, approve" natural language response Summary read aloud + mandatory "confirm" keyword. "Yes/yeah/sure" not accepted for high-stakes actions
Latency handling Typing indicator sufficient. Silence is expected and normal Filler phrases required ("I'm pulling that up..."). Silence >2s causes abandonment
Error / flag surfacing Visual warning badge + action menu with 4 options rendered as buttons Spoken warning + maximum 3 options read as a list. "Press 1" avoided — voice-native option naming
Audit reason collection Free text field with example shown inline as placeholder Tone prompt + open dictation. Example given verbally before the tone
Post-action state Confirmation message + reference number + "continue?" inline Confirmation spoken briefly. Reference number shortened (last 4 digits). Immediate continuation offer
Information density per turn High — full invoice card, PO details, vendor history, flags in one message Low — max 4 data points per utterance. Key number always spoken twice (currency amounts)
Error recovery User can scroll up, re-read, tap a different option at any point System must ask "Did you mean X or Y?" — no scroll back. Recovery prompts designed for voice re-entry
Escalation surface 4th button visible in UI at all times during flagged flows Escalation option only spoken if first 3 options are declined or user says "I'm not sure"
Session memory All 3 invoices tracked through session, user can jump to any Sequential only. Jumping out of order requires re-querying context — system handles gracefully

Key Design Insight

The largest divergence is not in the words used — it's in the confirmation architecture. Chat can rely on visual affordances (distinct button, summary card) to create confidence before action. Voice must encode that same confidence entirely in the acoustic signal — through the sequence of information spoken, the specific confirmation word required, and the absence of ambiguous response acceptance. Designing these the same way would be a critical failure.

Reusable Pattern Library

Six patterns extracted from this flow. Each is documented for reuse across other SAP Joule workflows. These are not tied to invoice approval — they apply to any high-stakes, multi-step enterprise AI task.

Pattern 01
Progressive Disclosure List
Trigger: User asks to see a list of items requiring action (invoices, tickets, requests)
Chat: Show full list with inline metadata
Voice: State count + highest priority only, offer sequential navigation
Failure: Listing all items on voice → cognitive overload, user re-asks
Pattern 02
High-Stakes Confirmation Gate
Trigger: Any irreversible action above $X threshold or affecting external party
Chat: Summary card + explicit "Yes, approve" required
Voice: Spoken summary + "confirm" keyword only — "yes" insufficient
Failure: Accepting "uh-huh" or Enter key as confirmation on financial actions
Pattern 03
Pre-Execution Flag Surface
Trigger: System detects anomaly, policy violation, or risk before action execution
Flag surfaces BEFORE confirmation — never after. User sees/hears the issue before being asked to decide.
Failure: Showing flag only in post-approval audit — user cannot course-correct
Pattern 04
Latency Bridge (Voice Only)
Trigger: API or data retrieval expected to exceed 1.5 seconds on voice channel
"I'm pulling that up..." / "Let me check that..." — filler before pause, not silence. Sets wait expectation without false completion signal.
Failure: Silence >2s → user repeats input, creates double-execution risk
Pattern 05
Structured Override with Audit Trail
Trigger: User requests to proceed despite system-flagged policy constraint
AI does not block override — but requires: (1) explicit reason stated, (2) reason logged in audit record, (3) relevant stakeholder notified automatically
Failure: Silently logging override without user awareness — erodes trust
Pattern 06
Graceful Escalation Path
Trigger: Task exceeds AI authority, user expresses uncertainty, or 2+ recovery attempts fail
Chat: Escalation always visible as 4th option on flagged flows
Voice: Offered only if user declines offered options or says "not sure"
Failure: Escalation buried in help menu — user gives up instead of routing to human

Metrics Framework & A/B Testing

Conversation design is measurable. These metrics define success for this flow and provide the basis for iterative improvement through experimentation.

Primary Success Metrics

Task Completion Rate
Approvals completed / Approval intents initiated
Target: >88% chat, >80% voice. Drop-off points tracked per dialogue node to identify where users abandon.
Turns to Completion
Avg utterances from intent to confirmed action
Target: ≤5 turns for standard approval. Flagged approvals: ≤8 turns. Optimization signal for over-clarification.
Confirmation Abandonment Rate
Users who reach confirmation but don't complete
High rate = confirmation friction. Particularly critical for voice "confirm" keyword pattern. Target: <5%.
Error Recovery Success
Users who hit error state and still complete
Target: >70%. Measures quality of error flow design vs. dead ends. Voice and chat tracked separately.
Escalation Rate
Tasks routed to human / Total tasks started
Target: <10% for standard approvals. High rate indicates AI confidence calibration issues or missing flow coverage.
Repeat Input Rate (Voice)
% of turns where user repeats their previous utterance
High rate = latency or comprehension issues. Directly measures latency bridge effectiveness. Target: <8%.

Responsible AI Design

Enterprise financial AI must be trustworthy by design — not as a compliance checkbox. These principles are embedded into the flow architecture, not applied as a post-design layer.

Trust Calibration
Joule never overstates confidence. When invoice data is incomplete ("goods receipt unconfirmed"), the system says so explicitly before surfacing approval — it does not silently proceed. Users must be able to trust that silence means all checks passed.
Hallucination Guardrails
Invoice amounts, PO numbers, and vendor names are never generated by the LLM — they are fetched from SAP's system of record and passed as structured data. LLM handles only language generation around confirmed facts. No critical financial data is inferred.
Human-in-the-Loop Architecture
The system is designed with explicit AI authority boundaries. Joule can approve standard invoices within policy. Flagged overrides always notify the controller — the AI surfaces the escalation need rather than making the compliance judgment itself.
Audit Trail Transparency
Every AI-assisted approval logs: the user, timestamp, invoice details, confirmation method (voice/chat), and for overrides, the stated reason. Users are told explicitly what is being logged and that it will appear in compliance reports.
Irreversibility Signaling
Before any action that cannot be undone, the system communicates consequence clearly: "This will trigger payment on the next ACH cycle." Voice includes this in spoken confirmation. Users are never surprised by the permanence of an action they confirmed.
No Pressure Architecture
Joule never creates urgency framing to drive approval ("this is overdue, approve now"). It surfaces factual status (7 days overdue) and lets the user decide. The AI facilitates decisions — it does not nudge toward outcomes.

Design Philosophy on AI Authority

The hardest responsible AI question in this flow: when should Joule refuse to proceed? The answer: Joule can always flag, always require explicit confirmation, and always surface a human path — but it should not refuse entirely. Refusal moves the work out of the auditable system into email or hallway conversations where it's harder to track. The AI's role is to make the right path easier and the risky path more deliberate — not to be a blocker.
