How a prompt earns its behavior

v1.0
The Naive First Draft
Written without constraints. See what the model does by default.
Prompt
You are a helpful assistant. Tell the user their invoice has a problem and ask what they want to do.
Model Output
Hi there! I noticed there seems to be an issue with one of your invoices. It looks like there might be a problem that needs your attention. Would you like to take a look at it and let me know how you'd like to proceed? I'm here to help!
Specificity: 1/5 Actionability: 1/5 Tone: 1/5 (too casual) Length: ok
Problem Diagnosis
Three critical failures:
1. No data injected — model has nothing to say so it generates vague filler ("seems to be an issue")
2. No role definition — "helpful assistant" gives the model no context about enterprise finance
3. No output structure — model defaults to conversational filler with zero actionable content

The output could apply to any invoice issue anywhere. It tells the user nothing and forces them to ask follow-up questions — the worst possible outcome for a time-pressured AP manager.
Missing: structured data Missing: role context Missing: output format
DIAGNOSIS APPLIED → ITERATION 2
v2.0
Inject Data + Define Role
Fix the two biggest structural gaps first.
Changes Made
Added: Structured invoice data injected as variables. Role defined as SAP finance assistant. Explicit instruction to name the specific issue.
Prompt v2
You are SAP Joule, an AI assistant for enterprise finance workflows. Invoice data: - Vendor: {{vendor_name}} - Amount: {{invoice_amount}} - PO limit: {{po_limit}} - Overage: {{overage_amount}} The invoice exceeds the PO limit. Tell the user what the specific problem is and ask what they want to do.
Model Output
This invoice from Global Supply for $231,000 exceeds your purchase order limit of $200,000 by $31,000. This needs to be resolved before approval can proceed. What would you like to do?
Specificity: 3/5 Actionability: 2/5 Tone: 3/5 Length: ok
Remaining Problems
Better — but still failing on actionability.
The user now knows the problem precisely. But "What would you like to do?" is an open question that forces the user to invent their own options. In an enterprise context, users need constrained, named choices — not a blank field.

Also: "needs to be resolved before approval can proceed" is passive and slightly alarming without context. An AP manager reading this cold may not know if this is a blocker, a warning, or a hard stop.
Fixed: specific amounts Fixed: role context Still missing: explicit options Still missing: severity framing
DIAGNOSIS APPLIED → ITERATION 3
v3.0
Add Constrained Options + Severity Signal
Give the user a decision menu, not a blank canvas.
Changes Made
Added: Explicit resolution options as a list. Severity framing ("this is a flag, not a hard block"). Instruction to format options clearly.
Prompt v3
You are SAP Joule, an AI assistant for enterprise finance. Invoice data: - Vendor: {{vendor_name}} - Amount: {{invoice_amount}} - PO limit: {{po_limit}} - Overage: {{overage_amount}} The invoice exceeds the PO limit. This is a policy flag — the user CAN still approve with an override, but must choose a resolution path. Present the issue clearly with the specific numbers. Then offer exactly these 3 options: 1. Request a PO amendment 2. Approve with override (requires reason) 3. Reject the invoice Do not suggest any other options. Do not ask open questions. Be direct and professional — this user is a finance professional.
Model Output
⚠ This invoice from Global Supply ($231,000) exceeds your PO limit of $200,000 by $31,000.

This is a policy flag — you can still proceed, but need to choose a resolution:

1. Request a PO amendment
2. Approve with override (you'll provide a reason)
3. Reject the invoice

Which would you like to do?
Specificity: 4/5 Actionability: 4/5 Tone: 4/5 Structure: 3/5
Remaining Problems
Significantly better. Two subtle issues remain:
1. "Which would you like to do?" at the end is redundant — the numbered list already implies a choice. It adds a sentence without adding meaning.

2. The warning emoji (⚠) was not instructed — the model added it. In an enterprise UI this might render inconsistently or be inappropriate depending on severity level. Visual markers should be controlled by the design system, not generated ad-hoc by the model.

3. Missing: escalation path. What if the user isn't sure? A 4th option (escalate to controller) is needed for compliance edge cases — but must be surfaced differently to avoid option overload.
Fixed: clear options Fixed: severity context New issue: uncontrolled emoji Missing: escalation path Redundant closing question
DIAGNOSIS APPLIED → ITERATION 4
v4.0
Control Output Format + Add Escalation
Lock down visual formatting. Add the escalation path without overloading the user.
Changes Made
Added: Explicit instruction to never use emojis or icons — those are rendered by the UI layer. Escalation option added as a secondary path, not a primary button. Removed closing question.
Prompt v4
You are SAP Joule, an AI assistant for enterprise finance. Invoice data: - Vendor: {{vendor_name}} - Amount: {{invoice_amount}} - PO limit: {{po_limit}} - Overage: {{overage_amount}} The invoice exceeds the PO limit. This is a policy flag — the user CAN still proceed with an override. Rules: - Never use emojis, icons, or special characters. The UI handles visual indicators. - Do not end with a question. The numbered list implies a choice. - Present exactly 3 primary options as a numbered list. - After the list, add one line: "If you're unsure, I can escalate this to your controller." Tone: Direct, professional. The user is a finance expert — do not over-explain.
Model Output
This invoice from Global Supply ($231,000) exceeds your PO limit of $200,000 by $31,000.

This is a policy flag — you can still proceed by choosing a resolution path:

1. Request a PO amendment
2. Approve with override (you'll provide a brief reason for the audit trail)
3. Reject the invoice

If you're unsure, I can escalate this to your controller.
Specificity: 5/5 Actionability: 5/5 Tone: 5/5 Format control: 4/5
One Remaining Edge Case
Almost production-ready. One edge case to handle:
What if the overage is very small (e.g. $50 on a $200K PO)? The current prompt treats all overages identically. A $50 overage likely warrants different language and possibly auto-approval under a threshold — this is a routing decision, not a prompt-only problem, but the prompt should not contradict the routing logic.

Also: "audit trail" in option 2 is helpful context but slightly jargon-heavy for a first-read. Test whether it improves or slows comprehension with real users.
Fixed: no emojis Fixed: escalation path Fixed: no redundant question Edge case: small overage routing

v5.0 ✓
Production Prompt — Conditional Logic + Threshold Handling
Add conditional behavior for overage severity. This is the prompt that ships.
Changes Made
Added: Conditional tone instruction based on overage percentage (minor vs. significant). Softened "audit trail" to "brief note." Minor cleanup of transition language.
Final Production Prompt
You are SAP Joule, an AI assistant for enterprise finance workflows. Invoice data: - Vendor: {{vendor_name}} - Amount: {{invoice_amount}} - PO number: {{po_number}} - PO limit: {{po_limit}} - Overage: {{overage_amount}} - Overage %: {{overage_pct}} The invoice exceeds the PO limit. Present this as a policy flag that the user can resolve. Conditional tone: - If overage_pct < 5%: Use neutral, matter-of-fact language. ("slight overage") - If overage_pct >= 5%: Use clear but calm flagging language. ("exceeds your PO limit by X") Format rules: - No emojis or special characters. UI layer handles visual indicators. - State the specific vendor, amount, and overage in the first sentence. - Present exactly 3 numbered options. - End with: "If you're unsure, I can escalate this to your controller." - Do not end with a question. Do not add any sentence after the escalation line. Tone: Direct and professional. This user is a finance expert.
Output — Large Overage (15%)
This invoice from Global Supply ($231,000) exceeds your PO #5501 limit of $200,000 by $31,000.

This is a policy flag — choose how to resolve it:

1. Request a PO amendment
2. Approve with override (you'll add a brief note for the record)
3. Reject the invoice

If you're unsure, I can escalate this to your controller.

Output — Small Overage (2%)
This invoice from Acme Corp ($204,000) has a slight overage of $4,000 against PO #4420-B ($200,000).

You can resolve this before approving:

1. Request a PO amendment
2. Approve with override (you'll add a brief note for the record)
3. Reject the invoice

If you're unsure, I can escalate this to your controller.
Specificity: 5/5 Actionability: 5/5 Tone: 5/5 Format control: 5/5 Edge cases: handled
Why This Prompt Works
Every sentence earns its place:
— Sentence 1: All critical data (who, how much, by how much) in one scan-friendly line
— Sentence 2: Reframes "problem" as "resolvable flag" — reduces anxiety, increases action rate
— Options 1–3: Complete decision space, no open questions, no invented paths
— Final line: Escalation path available without cluttering the primary options

What the prompt controls vs. what the UI controls:
The prompt controls language, structure, and conditional tone. The UI layer controls visual indicators (⚠ badge), button rendering, and option highlighting. This separation is intentional — prompts that generate their own UI elements become brittle when the design system changes.

Transferable Prompt Design Principles

Six lessons extracted from this iteration process. These apply to any LLM-powered enterprise conversation, not just invoice approval.

Lesson 01
Data in, data out
A prompt with no injected data produces vague output. Always structure what the model needs to know as explicit variables — never rely on the model to "figure it out."
Lesson 02
Constrain options explicitly
"What would you like to do?" forces users to invent solutions. Enterprise users need a named, bounded decision set — the model must be told to present exactly N options.
Lesson 03
Separate prompt layer from UI layer
Never let the model generate emojis, icons, or visual markers. These belong to the design system. Prompts that generate their own UI break when the system changes.
Lesson 04
Conditional tone is a prompt concern
The same factual situation at different severity levels deserves different language. Encode this logic in the prompt — don't rely on post-processing or a separate model call.
Lesson 05
Escalation as a designed path
Escalation to a human must be offered before the user gives up — not after 3 failed attempts. Design it as a visible exit at the bottom of every flag flow.
Lesson 06
End without a question
If the output already presents clear options, a closing question is redundant noise. Remove it. Silence after a decision menu is more confident than asking "what would you like to do?"
